# app-baseline-kit dependency

learning-management-system currently uses the **deprecated Pattern C**: `app-baseline-kit` is pinned to an exact Workflows SHA in `pyproject.toml` (`@13f94883220bbfb0ca69a4666fb44a49e0ae8172`, line ~77). This was a stopgap (commit 8f48efa) for a `uv` conflicting-URL error — `pyproject.toml` and `requirements.lock` used different URL forms for the same dependency, so `uv` refused to resolve; pinning pyproject to the exact SHA already in the lock made both sources agree and the install deterministic. The dependency-alignment test matches on package name (not URL) so it passes today, and the package ships `py.typed` (PR #2204). **This is the user's item-2 exception and must be migrated to Pattern A** — the no-emit approach eliminates the identical conflict class permanently without a frozen SHA. At test time the package is currently installed from the pinned SHA via `-r requirements.lock`.
