# LMS Comprehensive Audit — 2026-05-30

> Multi-agent audit of `learning-management-system`. 9 finders swept module clusters and two cross-cutting lenses (test-coverage, learning-science domain fidelity); every candidate finding was handed to an independent skeptic agent that tried to refute it against the real code. Only findings that survived refutation are listed.

**Run stats:** 65 agents · ~2.71M subagent tokens · 974 tool calls · ~15.9 min wall-clock.  
**Findings:** 55 raw → 55 deduped → **25 confirmed** after adversarial verification (11 flagged as high-confidence quick wins).

### ⚠️ Coverage caveat

The synthesis step and ~19 of the late verifier agents hit the account's usage cap near the end of the run. Those ~19 candidate findings were never adversarially verified and were **conservatively excluded** — so the 25 below are a *lower bound*. They're recoverable by re-running the verify phase after the cap resets (the workflow supports `resumeFromRunId`). This report was assembled deterministically from the verified findings, not by the (capped) synthesis agent.

## Executive summary

The codebase is feature-rich and the learning-science framing is reflected in real module structure, but the audit surfaced one **showstopper** and a small cluster of correctness/robustness issues that undercut core product promises:

- **The API is effectively unauthenticated.** A complete Argon2-based auth stack exists, and an auth dependency (`require_authenticated_user`) is defined — but it is attached to **zero endpoints**. Every route is reachable without credentials. This is the single most important thing to fix and should block any non-local deployment (note the in-flight Render deploy, issue #180).
- **The core "spacing" commitment is broken.** The spaced-repetition success ramp is frozen at step 0, so successful reviews are always rescheduled at ~1 day instead of expanding intervals. Spacing is a headline learning-science commitment in the README; today the scheduler doesn't actually space.
- **Content ingestion has correctness bugs.** The markdown importer attributes H2 sections to the wrong file's node in multi-file imports, and several importer paths crash on ordinary input (setext headings, non-UTF-8, invalid enums that pass dry-run then fail mid-import).
- **LLM cost controls are partly inert.** The operator-configurable daily cost cap is silently ignored in the production client path (hardcoded to 1,000,000 micro-USD), and budget preflight can under-estimate output cost.
- **Auth hardening gaps** beyond the wiring bug: username-enumeration timing side-channel, no password-strength policy, no login rate-limiting/lockout, unvalidated email.

Themes by the numbers — security and correctness dominate, and the two domain-fidelity findings both sit in the assessment/scheduling engine, the product's center of gravity.

#### require_authenticated_user is never wired to any endpoint — complete auth bypass in production
- **Severity:** critical  ·  **Category:** security  ·  **Confidence:** high  ·  **Location:** `src/lms/main.py:77-103`  ·  **Cluster:** Auth & security (src/lms/auth, src/lms/evidence, src/lms/research_registry)
- **What's wrong:** create_app() mounts every router (evidence/attempts, /inspect, /audit, /knowledge, /prompts, /capability, /llm, all /app/* UI pages, mastery, cases, scheduling, etc.) with NO authentication dependency. The session-aware guard require_authenticated_user (src/lms/auth/login.py:172) is the documented gate (its docstring and the comments in main.py:59,75 claim it 'actually gates downstream routes when AUTH_REQUIRED is true'), but a repo-wide grep shows it is referenced ONLY in main.py comments and in the test file — it is attached to zero real routes. No global middleware enforces it either: the only middleware is SessionMiddleware (main.py:60), which signs cookies but does not require them. Confirmed: every APIRouter is constructed without a `dependencies=` arg, main.py includes them without `dependencies=`, and tests/auth/test_authenticated_session.py builds its OWN throwaway FastAPI app and manually attaches Depends(require_authenticated_user) to test routes (lines 89-101) — so the passing tests give false confidence. render.yaml:32-33 sets AUTH_REQUIRED="true" for the deployed Render instance, so production runs in 'auth required' mode yet exposes the full learner-data API, audit log, and inspect/overview endpoints (e.g. GET /inspect/learners/{id}/overview returns mastery + recent evidence) to any anonymous visitor. The login form works but protects nothing.
- **Evidence:** main.py lines 83-103 include 20+ routers with no dependency; `rg require_authenticated_user src/` returns only comments in main.py and the def/docstring in login.py — no usage. Every router: `APIRouter(tags=[...])` with no dependencies=. render.yaml:33 `value: "true"` for AUTH_REQUIRED.
- **Fix:** Enforce the dependency app-wide. Simplest correct fix: in create_app(), add a router-level dependency on each protected include, e.g. define `auth_dep = [Depends(require_authenticated_user)]` and pass `app.include_router(attempts_router, dependencies=auth_dep)` (and same for inspect, audit, sources, graphs, prompts, feedback, cases, competencies, capability, mastery, review_queue, llm, and all ui routers), leaving login_router and health_router unguarded. Alternatively register a single global dependency via `FastAPI(dependencies=[Depends(require_authenticated_user)])` and exempt /login + /health inside the dependency. Add an integration test that runs the REAL create_app() with AUTH_REQUIRED=true and asserts an unauthenticated GET /inspect/... and GET /audit/events return 401/302, instead of the current test that builds a private app.
- **Verifier note:** Confirmed real and critical. Files: src/lms/main.py (create_app lines 77-106 — every include_router lacks dependencies=, FastAPI() at line 46 lacks global dependencies=); src/lms/auth/login.py:172 (require_authenticated_user — defined, documented, never wired); src/lms/api/inspect.py:22-26 and src/lms/api/audit.py:14-17 (protected data endpoints with only Depends(get_session)); render.yaml:32-39 (AUTH_REQUIRED="true", ENABLE_LOCAL_IDENTITY_ROUTES="false", no edge auth); tests/auth/test_authenticated_session.py:48-101 (builds throwaway app — 6 tests pass but never touch the real factory, hence false confidence). Empirically reproduced: anonymous GET /inspect/learners/{id}/overview and GET /audit/events both return HTTP 200 against the real create_app() with auth_required=True. Suggested fix in the report is sound: either pass dependencies=[Depends(require_authenticated_user)] on each protected include_router (leaving login_router + health_router open) or set FastAPI(dependencies=[...]) globally with /login and /health exempted, plus an integration test that runs the REAL create_app() with AUTH_REQUIRED=true and asserts 401/302 for anonymous protected requests.

#### Spaced-repetition success ramp is frozen at step 0 (always 1 day) because no code ever marks queue items 'completed'
- **Severity:** high  ·  **Category:** correctness  ·  **Confidence:** high  ·  **Location:** `src/lms/scheduling/service.py:247-261, 449-455, 575-582, 667-671`  ·  **Cluster:** Assessment, retrieval & scheduling engine (src/lms/scheduling, src/lms/feedback)
- **What's wrong:** The core spacing interval is chosen by `_success_interval_days(prior_successes)` over the ramp SUCCESS_INTERVALS_DAYS = (1, 3, 7, 14, 28). `prior_successes` comes from `_count_prior_successful_reviews`, which counts `ReviewQueueItem.status == "completed"` for due-review/new-learning items. However, NO production code path ever sets a ReviewQueueItem status to 'completed'. A repo-wide search shows the only 'completed' assignment is for a FeedbackAction (feedback/repository.py:404); there is no review-queue dispatch/complete endpoint (scheduling/api.py only exposes GET for the queue), and neither the attempt UI (ui/attempts.py) nor the demo touches queue status. I reproduced this with a real DB session: four consecutive successful reviews of the same node all emit rule='success-ramp-step-0' with prior_successes=0, so every successful review is rescheduled at 1 day forever. The 3/7/14/28-day expansion that defines spaced retrieval never happens for any real learner. The only test that exercises advancement (tests/scheduling/test_review_queue.py:186 `test_success_ramp_extends_after_prior_completions`) manually sets `first.status = 'completed'` at line 195, which masks the gap because production has no equivalent step.
- **Evidence:** service.py:257 `ReviewQueueItem.status == "completed"`; repro output: cycle 0..3 all `rule=success-ramp-step-0 prior_successes_seen=0`. Grep: only FeedbackAction gets status='completed'.
- **Fix:** Decide what 'a prior successful review' means and count it from a source production actually writes. Two viable fixes: (a) Add a queue-item completion transition (e.g., a POST /review-queue/{id}/complete route + repository helper that sets status='completed' and schedule_state='completed') and call it when a review attempt is recorded, so `_count_prior_successful_reviews` has real data; or (b) Re-base `_count_prior_successful_reviews` on a source that already exists for each successful retrieval — e.g., count prior EvidenceRecords for the (learner, knowledge_node) whose classified signal is 'success' (excluding the current attempt), or count prior completed/scheduled ReviewSchedule rows. Add an end-to-end test that runs 2-3 real schedule_from_attempt cycles (no manual status edit) and asserts due_at advances 1 -> 3 -> 7 days.
- **Verifier note:** Confirmed real, but downgraded from critical to high. It is a feature-correctness defect (spaced ramp permanently frozen at 1 day), not data loss/security/crash — items still schedule, just never expand. The QUEUE_STATUSES tuple defines "dispatched" and "completed" yet neither is ever written, indicating the review-completion lifecycle is unbuilt milestone scaffolding rather than a regression (the reporter's own fix (a) is "add a completion transition," i.e. new work). It remains high, not medium: this is the headline spaced-retrieval algorithm, the gap is invisible because the only advancement test fabricates the missing 'completed' state, and it directly undermines the Milestone-4 pre-registered day-30 retention gate (README:235). Single-user research project at Milestone 4-6 with no live learners caps real-world blast radius. Suggested fixes are sound; fix (b) (re-base on existing EvidenceRecords/ReviewSchedule rows) avoids inventing a new endpoint, and an end-to-end test asserting due_at advances 1->3->7 without manual status edits is the right regression guard.

#### Markdown importer links H2 sections to the wrong file's node (cross-file prerequisite edges)
- **Severity:** high  ·  **Category:** data-integrity  ·  **Confidence:** high  ·  **Location:** `src/lms/importers/markdown.py:133-146, 183, 196`  ·  **Cluster:** Curriculum, content &amp; importers
- **What's wrong:** In `_parse_heading_sections`, `parent_index = h1_index` is the index of the preceding H1 within that single file's heading list (markdown.py:183, 195-196). But `import_markdown_notes` accumulates `node_ids` GLOBALLY across all files and does `node_ids[heading.parent_index]` (markdown.py:133-134). When a directory import contains more than one file with headings, every H2's per-file parent index is misapplied to the global list, so prerequisite edges point at a node in a *different* file. Verified end-to-end: importing a.md (`# A1` / `## A2`) and b.md (`# B1` / `## B2`) produces edges `A1->A2` (correct) and `A1->B2` (WRONG — should be `B1->B2`); with three files, n.md's H2s (parent_index=0) all link to a.md's A1. The guard `heading.parent_index < len(node_ids)` (line 133) does not catch this because the global index is in range — it just points at the wrong node. This silently corrupts the imported knowledge graph with bogus prerequisite relationships. The existing test only imports a single file (tests/importers/test_markdown_importer.py:17-30), so it never exercises the cross-file path.
- **Evidence:** markdown.py:133 `if heading.parent_index is not None and heading.parent_index < len(node_ids):` then `create_knowledge_edge(... source_node_id=node_ids[heading.parent_index] ...)`. parent_index is set per-file at markdown.py:183 (`parent_index = h1_index if level == 2 else None`) where `h1_index = len(headings) - 1` is local to `_parse_heading_sections`.
- **Fix:** Make parent linkage file-local at creation time. Easiest: in `import_markdown_notes`, track node ids per file (e.g. reset a per-file index base when `heading.file_path` changes, or store `parent_index` as an offset within the current file and add the file's base offset). Alternatively, change `MarkdownHeading` to carry the parent's identity rather than a list index (e.g. resolve the parent heading object), so the edge source is unambiguous regardless of global ordering. Add a multi-file directory test asserting B2 links to B1, not A1.
- **Verifier note:** Verified by reproduction against a real in-memory DB session, not just static reading. Persisted edges came out as [('A1','A2'), ('A1','B2')] for a two-file directory import; correct would be [('A1','A2'), ('B1','B2')].

Severity held at high (not bumped to critical) because the corrupted records are explicitly draft-stage candidates intended for human review before promotion: nodes are status="draft"/provenance="imported", and edges are status="draft", confidence=0.5, with notes "Candidate prerequisite inferred from nested Markdown headings." (markdown.py:119-145). This mitigates blast radius somewhat — no authoritative/published graph data is directly written. But it is still silent, error-free cross-file data corruption of a core artifact (the knowledge graph) on a supported CLI feature with zero test coverage, so it clears the bar for high.

Suggested fix is sound: make parent linkage file-local. Cleanest option is to have MarkdownHeading carry the parent's identity (or a stable per-file key) rather than a bare list index, or in import_markdown_notes track a per-file base offset and add it to parent_index. Add a multi-file directory test asserting B2 links to B1, not A1. Note dry-run edge COUNT (line 93) is unaffected — only the source endpoint of created edges is wrong — so the bug is invisible in summaries and only shows up if you inspect edge endpoints.

#### FSRS adapter rates hint/reference-supported high-score answers as 'good' instead of 'hard' (rule ordering bug)
- **Severity:** medium  ·  **Category:** domain-fidelity  ·  **Confidence:** high  ·  **Location:** `src/lms/scheduling/fsrs_adapter.py:111-114, 143-150, 159-166`  ·  **Cluster:** Assessment, retrieval & scheduling engine (src/lms/scheduling, src/lms/feedback)
- **What's wrong:** FSRS_RULES is evaluated in order and the first match wins (evidence_to_fsrs_rating, line 188). The `partial-at-mastery` rule (returns 'good', value 3) is placed at index 3, BEFORE `supported-or-low-confidence-correct` (returns 'hard', value 2) at index 5. `_is_score_at_mastery` only checks `correctness is not False and score >= 0.85` and does NOT inspect hint_used / reference_accessed / support_level. So an answer that was correct with a revealed hint and normalized_score=0.95 is rated 'good'/value 3, even though the explicit intent (rule_id `supported-or-low-confidence-correct`, reason text 'Correct evidence with support ... maps to FSRS Hard') is to treat any supported correct answer as 'hard'. I confirmed this empirically: support_level='hint', hint_used=True, normalized_score=0.95 -> label='good', value=3, rule_id='partial-at-mastery'. In schedule_from_attempt this is partially masked because `_conservative_signal` blends FSRS with `_classify_signal` (which catches support_level) and downgrades to low-confidence-success, but the FSRS rating label/value/rule_id are still recorded in decision_log and the SchedulerDecision audit row (service.py:704-709), so the Inspect/explainability surface reports a wrong, over-confident rating.
- **Evidence:** Repro: `supported+mastery FSRS -> good 3 partial-at-mastery`. fsrs_adapter.py:113 `record.correctness is not False and score is not None and score >= 0.85` (no support check).
- **Fix:** Make the partial-credit rules support-aware or reorder so support is checked first for correct answers. Simplest: in `_is_score_at_mastery` (and ideally `_is_score_under_mastery`) add `and not _has_support(record)` so supported answers fall through to the `supported-or-low-confidence-correct` rule; or move the `supported-or-low-confidence-correct` and low-confidence checks ahead of the partial-credit-by-score rules in FSRS_RULES. Add a test: correctness=True, support_level='hint', normalized_score=0.95 must yield label='hard', rule_id='supported-or-low-confidence-correct'.

#### Username-enumeration timing side-channel in authenticate()
- **Severity:** medium  ·  **Category:** security  ·  **Confidence:** high  ·  **Location:** `src/lms/auth/repository.py:48-51`  ·  **Cluster:** Auth & security (src/lms/auth, src/lms/evidence, src/lms/research_registry)
- **What's wrong:** authenticate() returns None immediately when get_user_by_username finds no row, so the expensive Argon2id verify only runs for usernames that exist. The response-time difference (a DB lookup vs a DB lookup + ~tens of ms of Argon2 hashing) lets an attacker enumerate valid usernames despite the deliberately generic 'Invalid username or password.' message at login.py:145. The code comment at login.py:136-137 explicitly states the goal is to avoid username enumeration, but the timing channel defeats it.
- **Evidence:** repository.py:49 `if user is None: return None` runs before any hashing; verify_password (passwords.py:60) only executes for existing users. Argon2 default cost is ~tens of ms, easily distinguishable over the network.
- **Fix:** When the user is not found, perform a dummy Argon2 verify against a fixed throwaway hash before returning None, so both branches cost the same. E.g. compute a module-level `_DUMMY_HASH = PasswordHasher().hash("x")` once and call `verify_password(_DUMMY_HASH, password)` (discarding the result) in the user-is-None branch of authenticate(). Small, localized change in repository.py/passwords.py.

#### score_work_product can be re-run on any work product, creating duplicate transfer evidence and orphaning the prior score
- **Severity:** medium  ·  **Category:** data-integrity  ·  **Confidence:** high  ·  **Location:** `src/lms/cases/repository.py:326-422`  ·  **Cluster:** Core learning-object model (cases, competencies, mastery, capability)
- **What's wrong:** score_work_product() validates rubric presence, case existence, node resolution and score ranges, but never checks work_product.status. Calling it again on a product already in status 'scored', 'revision-requested', 'accepted' or 'withdrawn' unconditionally (a) creates a NEW Attempt + a NEW transfer EvidenceRecord via create_evidence_record (lines 379-404), (b) overwrites work_product.rubric_score_id with the new score, orphaning the previous RubricScore, and (c) forces work_product.status back to 'scored' (line 421), silently clobbering a 'revision-requested' or 'accepted' state. I confirmed this dynamically: scoring the same work product twice produced 2 EvidenceRecords and 2 RubricScores for the learner, with rubric_score_id pointing at the second and the first orphaned. Because every score emits a transfer EvidenceRecord into the same ledger that mastery/capability estimators read (mastery/service.py, capability/repository.py), repeated or accidental re-scoring inflates a learner's observed-evidence history and corrupts downstream mastery and capability scores. Tests only cover a single scoring (tests/cases/test_work_products.py:88 asserts exactly 1 evidence record), so the re-score path is unguarded and untested.
- **Fix:** At the top of score_work_product, guard the state transition: only allow scoring when work_product.status in ('submitted','draft') (or an explicitly allowed 'revision-requested' re-score path), else raise ValueError('work product is not in a scoreable state'). If re-scoring must be supported, first delete/supersede the existing EvidenceRecord and RubricScore referenced by work_product.rubric_score_id instead of leaving them orphaned, so the evidence ledger stays consistent.

#### Drift scan recomputes a whole-file hash when passage_range is non-line-range, producing false 'stale' results
- **Severity:** medium  ·  **Category:** correctness  ·  **Confidence:** high  ·  **Location:** `src/lms/sources/repository.py:279-290, 233`  ·  **Cluster:** Curriculum, content &amp; importers
- **What's wrong:** `_select_passage` silently returns the ENTIRE file text when `passage_range` does not match `_LINE_RANGE_PATTERN` (repository.py:283-284: `if match is None: return text`). The pattern only accepts numeric line ranges like `L2-L5`/`2-5`. A SourceReference created with passage-level `content` plus a human-readable range (e.g. `"Section 1"`, `"Chapter 2"`, `"pp. 5-10"` — all valid per the 120-char free-text `passage_range` field in schemas.py:21 and models.py:60) stores a hash over the passage content, but the FIRST `scan_source_references` pass recomputes a whole-file hash and flips `drift_status` to `stale` even though nothing changed. Verified: created a markdown-file ref with `passage_range='Section 1'` + `content='Key passage about X.'`; an immediate scan of the UNCHANGED file marked it `stale` (stored passage hash 4fb15be3 vs recomputed whole-file hash 60c961d7). A related variant: `update_source_reference` only recomputes the hash when `content` is supplied (repository.py:138-145), so a PATCH that changes `passage_range` (or `stable_locator`) without re-sending content leaves `content_hash` keyed to the old passage — verified to also yield a false `stale` on the next scan. Both undermine the trustworthiness of the entire drift-detection feature: operators get 'this prompt may be built on stale source material' alerts that are noise.
- **Evidence:** repository.py:282-284 `match = _LINE_RANGE_PATTERN.match(passage_range.strip())` / `if match is None:` / `return text`. Scan comparison at repository.py:233 `new_status = "current" if new_hash == reference.content_hash else "stale"`.
- **Fix:** Do not silently hash the whole file on an unparseable range. Either (a) raise a ValueError in `_select_passage` when `passage_range` is set but unparseable so the inconsistency surfaces at create/scan time, or (b) treat an unparseable range as 'whole file' consistently at BOTH create and scan time. Also, in `update_source_reference`, recompute `content_hash` whenever `passage_range` or `stable_locator` changes for a markdown-file reference (mirror the create-time logic) instead of only when `content` is passed.

#### Drift detection silently skips all non-markdown source types (tracked as #182), making drift undetectable for most documented types
- **Severity:** medium  ·  **Category:** correctness  ·  **Confidence:** high  ·  **Location:** `src/lms/sources/repository.py:215-217, 195-202, 22-30`  ·  **Cluster:** Curriculum, content &amp; importers
- **What's wrong:** `scan_source_references` buckets every non-`markdown-file` reference into a single opaque `skipped` count and continues (repository.py:215-217). The schema enumerates five source types (markdown-file, kindle-highlight, url, pdf-passage, internal-note — models.py:17-23), but a changed `internal-note` or `url` can never surface as `drift_status == "stale"`. `compute_source_hash_for_reference` reinforces this by raising ValueError for every non-file type (repository.py:202). `internal-note` content is actually persisted (the CSV importer stores JSON content via `_row_content`, csv_graph.py:198-199), so it is hashable today with no network — yet it is skipped. The single `skipped` field (DriftScanSummary, repository.py:22-30) also conflates 'deliberately unsupported' (pdf/kindle) with 'no-op', so operators cannot distinguish a coverage gap from nothing-to-do. This is exactly the gap issue #182 documents and asks to close (internal-note hashing, opt-in url fetch, per-reason counts).
- **Evidence:** repository.py:215 `if reference.source_type != "markdown-file": skipped += 1; continue`; repository.py:202 `raise ValueError("content or content_hash is required for non-file source references")`.

Note: stored note content is not currently persisted on SourceReference (only the hash is), so the `internal-note` path needs a content column or a way to re-derive content; scope that as part of the fix.

#### Setext-style Markdown headings crash the importer with an unhandled ValueError
- **Severity:** medium  ·  **Category:** error-handling  ·  **Confidence:** high  ·  **Location:** `src/lms/importers/markdown.py:251-267`  ·  **Cluster:** Curriculum, content &amp; importers
- **What's wrong:** `_extract_h1_h2_headings` uses Mistune's AST parser, which recognizes setext headings (`Title` underlined with `===` for H1 or `---` for H2 — valid CommonMark). But `_find_heading_line` only matches ATX (`#`-prefixed) headings via `_ATX_HEADING_RE` (markdown.py:17, 259), so it cannot map a setext heading to a source line and raises `ValueError("could not map parsed heading to source line")` (markdown.py:267). #### Operator-configured daily cost cap (LLM_DAILY_CAP_MICRO_USD) is silently ignored; production client hardcodes 1_000_000
- **Severity:** medium  ·  **Category:** data-integrity  ·  **Confidence:** high  ·  **Location:** `src/lms/llm/api.py:183-198`  ·  **Cluster:** LLM provider layer & prompts
- **What's wrong:** _default_client() loads the env-driven cap into LLMConfig.global_daily_cap_micro_usd (api.py:188, from load_llm_config_from_env which reads LLM_DAILY_CAP_MICRO_USD, config.py:74-75), but the object that actually enforces the kill-switch is DailyBudgetTracker, and it is constructed with a hardcoded global_cap_micro_usd=1_000_000 (api.py:197). budget.preflight() only ever checks DailyBudgetTracker.global_cap_micro_usd (budgets.py:36); LLMConfig.global_daily_cap_micro_usd is never read for enforcement anywhere (confirmed via grep). So an operator who sets LLM_DAILY_CAP_MICRO_USD to lower spend gets no effect: the real cap stays at the hardcoded 1_000_000. Likewise per-mode caps are always empty (mode_caps_micro_usd={}), so per-mode budgeting never fires either. This defeats the project's documented fail-closed cost kill-switch for the deployed client.
- **Fix:** In _default_client(), pass the env-derived cap into the tracker: DailyBudgetTracker(mode_caps_micro_usd=env_config.per_mode_daily_cap_micro_usd, global_cap_micro_usd=env_config.global_daily_cap_micro_usd). Remove the hardcoded 1_000_000. Optionally drop the now-redundant global_daily_cap_micro_usd field from LLMConfig or wire LLMClient to read it, so there is a single source of truth for the cap.

#### No error handling or retry around the real Anthropic call; transient/rate-limit/timeout errors surface as unhandled 500s
- **Severity:** medium  ·  **Category:** error-handling  ·  **Confidence:** high  ·  **Location:** `src/lms/llm/providers.py:164-215`  ·  **Cluster:** LLM provider layer & prompts
- **What's wrong:** AnthropicProvider.complete() calls client.messages.create(**call_kwargs) (providers.py:182) with no try/except, no retry, and no backoff (confirmed: no retry/tenacity/backoff/APIError/RateLimit handling anywhere in src/lms/llm). Any anthropic.APIError, RateLimitError (429), APITimeoutError, or connection error propagates raw out of the wrapper as a non-LLMError exception. The only real-provider call site, authoring_assist_propose_route (api.py:331-359), catches only (SourceConstraintViolation, ValueError), so a provider/network failure becomes an uncaught exception -> HTTP 500 with a stack trace, rather than a controlled 502/503. There is also no automatic retry for transient 429/5xx, so normal rate limiting fails the user-facing request outright.
- **Fix:** Wrap the create() call in AnthropicProvider.complete() in try/except for anthropic SDK errors and re-raise as a new LLMError subclass (e.g. ProviderCallError) so callers can catch a single wrapper exception type. Add bounded retry with exponential backoff for RateLimitError/APITimeoutError/5xx (either via the SDK's max_retries on the Anthropic client or an explicit loop). Update authoring_assist_propose_route to catch the new wrapper error and return 502/503.

#### DailyBudgetTracker is mutated without locking under FastAPI's sync-route threadpool; concurrent calls can overspend the cap
- **Severity:** medium  ·  **Category:** correctness  ·  **Confidence:** high  ·  **Location:** `src/lms/llm/budgets.py:25-59`  ·  **Cluster:** LLM provider layer & prompts
- **What's wrong:** All LLM routes are sync `def` (api.py:331,493, etc.), so FastAPI executes them in a threadpool. _default_client() is @lru_cache(maxsize=1) (api.py:168), so every concurrent request shares one DailyBudgetTracker instance. preflight() reads self._global_spend and record() does self._global_spend += cost (budgets.py:35-36,59) with no lock. This is a non-atomic read-modify-write across threads: two requests can both pass preflight against the same pre-call _global_spend and then both record(), pushing actual spend past global_cap_micro_usd. The kill-switch's entire purpose (fail-closed before money is spent) is undermined precisely when traffic is high. Lost updates on _mode_spend/_global_spend can also under-count recorded spend.
- **Fix:** Guard preflight+record with a threading.Lock held across the read-and-update (or make preflight reserve the projected cost atomically and reconcile in record). Simplest: add a threading.Lock to DailyBudgetTracker and acquire it in preflight() and record(); ideally expose a single reserve/commit API so the check and the spend update are one critical section.
